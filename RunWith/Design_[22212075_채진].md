## 1. Introduction

러닝은 특별한 장비 없이 언제 어디서든 할 수 있어서 꾸준히 하는 사람들이 많다. 그런데 혼자 뛰다 보면 금방 지루해지거나 페이스를 유지하기 힘든 경우가 많다. 같이 뛸 사람이 있으면 훨씬 더 오래 달릴 수 있다는 걸 알면서도, 막상 비슷한 페이스와 일정을 가진 러닝 파트너를 찾는 것이 쉽지 않다. 러닝 크루에 들어가자니 부담스럽고, 친구에게 부탁하자니 페이스가 안 맞는 경우도 많다.

아래는 Analysis에 이은 이 시스템 개발의 세 번째 단계인 Design에 관한 내용으로써, 실제 시스템 구현에 직접적으로 관여하는 모든 요소들의 윤곽을 확정하고 구체적으로 디자인 해 나가는 내용을 다루고 있다. 본 문서의 모든 세부 사항은 직접적인 구현 시 소스코드상에서의 용례와 완벽히 일치함을 목적으로 한다.

| **Report** | 신고 정보를 저장하는 클래스이다. 차단은 User 클래스의 blockList를 통해 관리된다.<br><br>- reportId : int — 신고 고유 번호<br>- reporterId : String — 신고자 ID<br>- targetUserId : String — 신고 대상 사용자 ID<br>- reason : String — 신고 사유<br><br>- submitReport() : void — 신고를 접수하는 메소드이다.<br>- blockUser(targetUserId : String) : void — 특정 사용자를 User의 blockList에 추가하는 메소드이다. |

## 4. State machine diagram

아래의 그림은 RunWith 시스템의 State machine diagram을 표현한 그림이다.

<img width="442" height="582" alt="image" src="https://github.com/user-attachments/assets/7e830412-2a2b-4838-89b7-237d1883bf3a" />

아래는 위의 State Machine Diagram에 나온 각 State들에 대해서 간단하게 설명한 표이다.

| Status | Explanation |
|--------|-------------|
| Launch System | 앱이 실행된 초기 상태이다. |
| Sign Up / Log In | 회원가입 또는 로그인 화면이 표시된 상태이다. 신규 사용자는 회원가입 흐름으로, 기존 사용자는 로그인 흐름으로 진입한다. |
| Wait Login Verification | 입력한 이메일과 비밀번호가 맞는지 서버에서 검증하는 상태이다. true이면 메인 화면으로, false이면 다시 로그인 화면으로 돌아간다. |
| Register Profile | 회원가입 완료 후 프로필이 없는 경우 프로필 등록 화면이 표시된 상태이다. 페이스, 선호 거리, 시간대, 자기소개를 입력한다. |
| Main (Home) | 로그인 완료 후 메인 화면 상태이다. 내 러닝 프로필 요약, 예정된 약속, 빠른 파트너 검색을 확인할 수 있으며 모든 기능을 이 상태에서 시작한다. |
| Search Partner | 페이스, 거리, 시간대 조건을 설정하고 근처 러너 목록을 조회하는 상태이다. |
| View Partner Profile | 검색 결과에서 특정 파트너의 페이스, 선호 거리, 자기소개, 신뢰도 점수를 조회하는 상태이다. |
| Wait Match Response | 매칭 요청을 보낸 후 상대방의 수락 또는 거절을 기다리는 상태이다. |
| Matched | 매칭이 성사된 상태이다. 채팅방이 자동으로 생성되며 이 상태에서 채팅을 시작할 수 있다. |
| Chat | 매칭된 파트너와 메시지를 주고받는 상태이다. 채팅 중 약속 등록이 가능하다. |
| Create Appointment | 채팅 중 약속 등록 버튼을 눌러 날짜, 시간, 출발 장소를 입력하여 약속을 생성하는 상태이다. |
| Appointment List | 등록된 약속 목록을 확인하는 상태이다. 약속 완료 버튼을 눌러 Confirm Complete 상태로 이동할 수 있다. |
| Confirm Complete | 러닝이 끝난 후 약속 화면에서 완료 버튼을 눌러 약속을 완료 처리하고 러닝 이력을 기록하는 상태이다. |
| Write Review | 약속 완료 확인 후 자동으로 표시되는 화면으로, 파트너에 대한 평점과 한 줄 리뷰를 작성하는 상태이다. |
| Edit Profile | 등록된 프로필(페이스, 거리, 시간대, 자기소개 등)을 수정하는 상태이다. |
| Report / Block | 불쾌한 사용자를 신고하거나 차단하는 상태이다. 차단된 사용자는 이후 파트너 검색 결과에서 제외된다. |

## 5. Implementation requirements

RunWith 시스템을 구동하기 위해 필요한 요구사항은 아래와 같다.

1) Hardware Requirements

- CPU : Intel PENTIUM Ⅳ 이상 (또는 ARM 기반 모바일 AP)
- RAM : 2GByte 이상
- Storage : 100MByte 이상의 여유 공간
- Network : 인터넷 연결 필요 (매칭, 채팅, 푸시 알림)

2) Software Requirements

- OS : Android 9.0 이상 / iOS 13.0 이상
- Implementation Language : JavaScript (React Native + Expo)
- Map API : Google Maps API (위치 기반 매칭)
- Push Notification : Firebase Cloud Messaging (FCM)
- Database : Firebase Firestore (사용자 정보, 채팅, 매칭 이력)
- Authentication : Firebase Authentication

3) Nonfunctional Requirements

- Response Time : 주요 기능 응답 시간 3초 이내
- Security : 비밀번호 암호화 저장, HTTPS 통신
- Concurrency : 다수 사용자 동시 접속 지원

## 6. Glossary

용어사전에 대해선 다음의 표와 같다.

| Terms | Description |
|-------|-------------|
| Class Diagram | 객체지향형 시스템 설계에서, 시스템의 논리 설계를 위한 클래스들의 존재와 그들의 관계를 도식으로 정의한 것. 단일 클래스 다이어그램은 시스템 클래스 구조를 보여줌. |
| Sequence Diagram | 시스템의 동작을 시간 순서에 따라 객체 간 상호작용으로 표현한 다이어그램이다. |
| State Machine Diagram | 상태 다이어그램은 컴퓨터 과학 및 관련 분야에서 시스템의 동작을 설명하는 데 사용되는 다이어그램의 유형이다. 설명된 시스템이 한정된 수의 상태로 구성되어야 한다. |
| boolean | 컴퓨터 언어에 있는 데이터 유형으로써 true와 false 두 가지 결과를 가진다. |
| 메소드 | 멤버 함수라고도 하며, 객체지향 프로그래밍 언어에서 클래스 혹은 객체에 소속된 서브루틴을 가리킨다. |
| FCM | Firebase Cloud Messaging. Google이 제공하는 크로스 플랫폼 푸시 알림 솔루션이다. |
| MatchingEngine | 사용자 조건(페이스, 거리, 위치 등)을 기반으로 적합한 파트너를 탐색하고 필터링하는 로직 컴포넌트이다. |
| 매칭 (Matching) | 두 사용자가 서로 러닝 파트너로 연결되는 것 |
| 페이스 (Pace) | 1km를 달리는 데 걸리는 시간. 예: 6분/km |
| 신뢰도 점수 | 리뷰와 약속 이행 여부를 바탕으로 계산되는 사용자의 신뢰 지표 |
| 노쇼 (No-show) | 러닝 약속을 잡고 나타나지 않는 행위 |
| 차단 | 특정 사용자와 더 이상 매칭되지 않도록 User의 blockList에 추가하는 기능 |
| DateTime | 날짜와 시간 정보를 함께 저장하는 데이터 타입이다. |
| extend | UML Use Case 관계의 일종으로, 특정 조건에서 기본 Use Case의 흐름에 추가 기능이 덧붙여지는 관계이다. |
| include | UML Use Case 관계의 일종으로, 기본 Use Case가 반드시 포함하는 다른 Use Case를 가리키는 관계이다. |
